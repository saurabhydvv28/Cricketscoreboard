# Project Memory: Local Cricket Scoreboard

## Project Overview
A full-stack Local Cricket Match Scoreboard web application built with Next.js 14 (App Router), Tailwind CSS, hand-built shadcn-style components, and Supabase (PostgreSQL + Auth + Realtime + RLS). The app supports two user roles: public viewers/players who can register and view live scores/leaderboards, and admins who can create matches and score ball-by-ball in real-time. All core features are now implemented: scaffolding, design system, auth, admin match creation, ball-by-ball scoring console, and real-time public scoreboard. Only the leaderboard dashboard (Step 7) remains.

## System Configuration & Tech Stack
- **Frontend:** Next.js 14.2.35 (App Router), TypeScript, Tailwind CSS v3, hand-built shadcn-style components (Radix UI primitives + cva), Lucide React icons
- **Backend/DB:** Supabase (PostgreSQL, Auth, Realtime, RLS) via `@supabase/ssr@0.12.0` + `@supabase/supabase-js@2.108.0` (cookie-based SSR auth) — versions pinned exact, see edge case note below
- **Fonts:** Geist Sans (display/UI) + Geist Mono (scoreboard ticker digits) — bundled locally as variable fonts, no external font CDN needed
- **Design theme:** "Stadium Ledger" — deep pitch-green/navy dark theme, stump-LED amber accent, boundary-red for wickets/sixes, tabular-mono scoreboard numerals. See `src/app/globals.css` for tokens.
- **Package Manager:** npm

## Discovered Edge Cases & Technical Nuances
- PostgreSQL views for Leaderboards require explicit `CAST` to avoid BigInt hydration errors in Next.js SSR — all aggregated integer fields cast as `INTEGER` or `NUMERIC` in the views.
- `live_data` JSONB field on `matches` stores volatile state (striker_id, non_striker_id, bowler_id, current_score, wickets, overs_bowled, current_over_balls) for O(1) scoreboard rendering without expensive joins on every ball.
- Wide and No Ball deliveries do NOT count as legal balls (ball_number is not incremented for the over in those cases); the `ball_by_ball_logs` table records a `is_legal_delivery` boolean to handle this.
- Strike rotation: strike changes on odd legal runs scored by batsman; strike always changes at end of over.
- `player_id` on profiles is a human-readable slug (e.g., `virat-k-42`) generated at signup for easy team assembly by admins.
- RLS on `ball_by_ball_logs` and `matches` for write operations checks `profiles.is_admin = true` via a security-definer function to avoid RLS recursion.
- **shadcn CLI registry (`ui.shadcn.com`) is unreachable on this sandbox's network allowlist** — all shadcn-style components must be hand-built from `@radix-ui/react-*` primitives + `class-variance-authority` instead of `npx shadcn add <component>`. The `@radix-ui/react-badge` package does not exist on npm at all — Badge was built as a plain styled div with cva variants, no Radix primitive needed.
- `next/font/google` will fail at build time on this sandbox (Google Fonts CDN not in network allowlist) — use `next/font/local` with the Geist woff files already bundled by `create-next-app`, or self-host any other fonts under `src/app/fonts/`.
- `@supabase/supabase-js` triggers an Edge Runtime warning during build ("A Node.js API is used... process.version") when imported transitively into `middleware.ts` via `@supabase/ssr`. This is benign — middleware only exercises the auth/cookie subset of the client — but worth knowing it's expected noise in build output, not a real error.
- Tailwind v3 (not v4) was scaffolded by `create-next-app` in this environment — config lives in `tailwind.config.ts` with a traditional `content` array, not the v4 CSS-first `@theme` approach.
- Design tokens use HSL channel triples in CSS vars (e.g. `--background: 152 28% 6%`) consumed via `hsl(var(--x))` in Tailwind config, the standard shadcn convention — keeps dark theme + future light theme variants easy to add later.
- **CRITICAL TypeScript gotcha — `interface` vs `type` in Database row definitions completely breaks Supabase query type inference.** Discovered while building Step 3: every `.from(table).select(...)` result was silently typed as `never` (e.g. `Property 'is_admin' does not exist on type 'never'`), with no error pointing at the real cause. Root cause: `@supabase/supabase-js@2.108.x`'s `GenericTable` constraint requires `Row`/`Insert`/`Update` to structurally satisfy `Record<string, unknown>`. TypeScript `interface` declarations do NOT satisfy open index-signature constraints like this even when all declared properties match — only `type` object-literal aliases do, because interfaces support declaration merging and TS won't implicitly grant them an index signature. **Fix: every row/table shape in `src/types/database.ts` (`Profile`, `Match`, `BallByBallLog`, all leaderboard view rows) must be declared with `type X = { ... }`, never `interface X { ... }`.** `OrangeCapRow`/`PurpleCapRow` use `BattingStatsRow & { rank: number }` intersection instead of `interface X extends Y` for the same reason. If adding new tables/views to the schema in future steps, keep this pattern.
- Also required for correct inference: the `Database` type needs a `__InternalSupabase: { PostgrestVersion: '12' }` marker (matches current official `supabase gen types typescript` output format), and every `Tables`/`Views` entry needs an explicit `Relationships: []` field — both are part of the `GenericTable`/`GenericSchema` constraint in supabase-js 2.108.x and are silently required even though older supabase-js versions didn't need them.
- `@supabase/ssr` and `@supabase/supabase-js`/`@supabase/postgrest-js` versions must stay in sync (lockfile can drift these out of step during `npm install`, which can also surface as `never` types) — currently pinned: `@supabase/ssr@0.12.0`, `@supabase/supabase-js@2.108.0`.
- React 18 / Next.js 14.2 form state pattern: use `useFormState` + `useFormStatus` from `react-dom`. Do NOT use `useActionState` (that's the React 19 rename) — it doesn't exist in this project's React version and would fail to import.
- Server actions that call `redirect()` (e.g. `signup`, `login`, `logout`) must NOT be wrapped in try/catch that swallows the redirect — `redirect()` works by throwing a special Next.js internal error that must propagate. The auth actions in `src/lib/actions/auth.ts` return early instead of redirecting on validation/auth errors, and only call `redirect()` on the success path, after `revalidatePath`.
- Admin route protection is now two-layered: `src/middleware.ts` redirects unauthenticated `/admin/*` requests to `/login` (cheap, no DB query), and `src/app/admin/page.tsx` does the real `profiles.is_admin` check server-side and renders an "Admins only" message for authenticated non-admins (can't be done cheaply in middleware without a DB round-trip on every request).
- **Vercel `MIDDLEWARE_INVOCATION_FAILED` (500 on every route) — caused by `process.env.X!` non-null assertions in the Supabase client setup.** If `NEXT_PUBLIC_SUPABASE_URL`/`NEXT_PUBLIC_SUPABASE_ANON_KEY` aren't set in the deployment platform's environment variables (e.g. forgotten in Vercel Project Settings → Environment Variables), the old code called `createServerClient(undefined!, undefined!, ...)`, which throws inside middleware — and since middleware runs on every single request, this took down the entire site with no useful error message. **Fix applied:** added `src/lib/supabase/env.ts` with a `getSupabaseEnv()` helper that throws a clear, actionable error message ("Set NEXT_PUBLIC_SUPABASE_URL...") instead of relying on `!`. The browser client (`client.ts`) and server client (`server.ts`) both now use this helper and will throw a readable error if misconfigured. **Middleware is the one exception** — `src/lib/supabase/middleware.ts` intentionally does NOT use the throwing helper; instead it checks the env vars manually and, if missing, logs an error and lets the request continue unauthenticated rather than crashing every route on the site. This means a misconfigured deployment now degrades to "auth doesn't work, but the site loads" instead of "the entire site 500s with no explanation." Verified by building and running `next start` with zero env vars set (homepage still 200s; auth-dependent pages throw a clear, readable error in server logs).
- **Vercel Output Directory error ("No Output Directory named 'public' found")** is unrelated to anything in the codebase — there's no `vercel.json` in this project overriding it. This happens when the Vercel project's **Framework Preset** is set to something other than "Next.js" (e.g. "Other" or a static-site preset), causing Vercel to look for a static `public/` build output instead of running `next build` and serving `.next/`. Fix is in Vercel dashboard → Project Settings → Build and Deployment → Framework Preset → set to "Next.js"; also verify Root Directory points at the folder containing `package.json` if the zip was extracted into a nested folder.
- **`README.md` was silently overwritten during Step 2 scaffolding.** The custom README written in Step 1 got clobbered when `create-next-app` ran (it generates its own boilerplate README), and this went unnoticed until Step 3+ cleanup, since the file still existed and had a plausible-looking diff size. Restored with full project docs + deployment troubleshooting in this session. **Lesson for future steps: after running any scaffolding tool (`create-next-app`, `shadcn init`, etc.) that writes to the project root, always re-check `memory.md` and `README.md` weren't overwritten before continuing** — these aren't part of the tool's expected output but can still get clobbered if they collide with generated boilerplate filenames.
- **`PlayerPicker` is a controlled component, not self-managed state** — this was a deliberate design choice, not the obvious first approach. Team A and Team B each render their own `PlayerPicker` instance; if each managed its own `selected` state internally, neither would know what the other had picked, so a player could be added to both rosters with no warning until form submission (server-side validation would catch it, but only after a full round-trip). Lifting state to `CreateMatchForm` and passing `selected`/`onChange`/`excludeIds` down means the moment a player is added to Team A, they instantly disappear from Team B's search results — no submit needed to discover the conflict.
- **Server actions that need to be called directly from event handlers (not `<form action={...}>`) should use `useTransition`, not `useFormState`.** `TossForm` and `DeleteMatchButton` both need multiple pieces of client state confirmed before calling a server action (toss winner AND decision; delete confirmation step) — `useFormState` only fires on form submission, so these call the server action directly inside a `startTransition()` callback instead, which still gives a `isPending` flag for loading states without requiring an actual `<form>` wrapper.
- **`formData.getAll(fieldName)` is the correct way to receive an array of values from multiple same-named inputs** — `PlayerPicker` renders one `<input type="hidden" name="teamARoster" value={playerId}>` per selected player, and `createMatch` reads them all at once via `formData.getAll('teamARoster') as string[]`. This avoids needing to JSON-stringify the roster array into a single hidden input and parse it server-side.
- **`requireAdmin()` in `matches.ts` is a defense-in-depth layer, not the only protection.** RLS policies from Step 1 (`matches_write_admin`, etc.) already enforce `is_admin()` at the database level — even if `requireAdmin()` were accidentally removed from a server action, a non-admin's insert/update/delete would still be rejected by Postgres. The application-level check exists purely so non-admins see a clean error message/redirect instead of a raw Postgres permission-denied error bubbling up.
- Match status transitions are intentionally one-directional from the UI: `scheduled → live` (via `startMatch`, sets toss + flips status) and matches stay `live` until Step 5's scoring console marks them `completed` (not yet implemented). `deleteMatch` is scoped with `.eq('status', 'scheduled')` so it's a no-op (silently deletes 0 rows) if someone races a delete against a match that just went live — this relies on Postgres row-level atomicity rather than a separate check-then-delete, avoiding a TOCTOU race.
- **A single `runsRun` field drives strike rotation correctly for every delivery type, including byes and leg-byes.** `BallInput.runsRun` is deliberately NOT "runs off the bat" — it's "runs physically run between the wickets," full stop. For a normal delivery these are the same thing. For a bye/leg-bye, the bat didn't touch the ball but the batsmen still ran, and that's what should drive rotation — so the UI's extras dialog asks "how many runs were run" for byes/leg-byes and feeds that directly into `runsRun`. For a wide/no-ball with no run attempted, `runsRun` is 0 even though `extraRuns` (the score impact) is at least 1. This single-field design avoids needing separate "runs off bat" vs "byes run" vs "extras run" fields with their own odd/even rotation rules that would have to be kept in sync.
- **Non-run-out wickets do NOT rotate strike even if `runsRun` happens to be nonzero.** `shouldRotateStrike()` explicitly short-circuits to `false` whenever `isWicket && wicketType !== 'run_out'` — a batsman given out bowled/caught/LBW/etc. didn't complete a run by definition (the ball is dead the moment the dismissal happens), so even though the UI's wicket dialog only shows a runs-input for run-outs specifically (the other 8 dismissal types don't ask for a run count at all, so `runsRun` defaults to 0 for them anyway), the rotation logic doesn't trust that — it checks `wicketType` directly so it's correct even if a future UI change accidentally passed a nonzero value for a non-run-out dismissal.
- **CRITICAL — the order of operations in `computeNextLiveData` for run-out wickets is rotate-then-clear, and this is correct even though it looks backwards on first read.** The dismissed-batsman-clearing check (`if (input.playerDismissedId === striker) striker = null`) runs AFTER strike rotation has already been applied for the ball. This matters specifically for run-outs: if N runs were completed before the run-out, the two batsmen have already physically crossed N times, so whoever's name was passed as `playerDismissedId` may now be standing at the opposite named end from where they started the ball. The check is against the POST-rotation striker/non-striker, which correctly identifies which end's slot to clear based on where that named player actually ended up standing when given out — not which role (striker/non-striker) they held at the start of the delivery. **Hand-traced and verified**: striker=A, non-striker=B, 1 run completed then B is run out → rotation swaps to striker=B/non-striker=A (since 1 is odd) → the clear-check sees `playerDismissedId === 'B'` matches the (now-rotated) striker slot → clears striker, correctly leaving A as non-striker. This was initially mistaken for a bug while verifying (the naive expectation is "B should just be cleared from wherever B started," i.e. non-striker) — but that naive expectation is the actual error; the rotate-then-clear order matches physical reality. **The wicket dialog's UI copy was deliberately written to remove the adjacent ambiguity**: it asks for "runs fully completed before the dismissal" and explicitly says "don't count the run in progress when the fielder broke the stumps," since a run-out by definition happens mid-run and the count must reflect only runs that were safely completed, not the interrupted one.
- **`undoLastBall` reconstructs `live_data` by deleting the most recent ball log row and replaying every remaining ball in the innings from scratch through `computeNextLiveData`**, rather than attempting to algebraically reverse a single ball (which would require correctly un-rotating strike, un-incrementing overs, potentially un-clearing a dismissed batsman, etc. — all error-prone to invert directly). The replay seeds the opening striker/bowler from the first remaining ball's `batsman_id`/`bowler_id` columns (which the schema already records per-ball), and recovers the opening non-striker by scanning for the second distinct `batsman_id` to appear in the log — correct for the overwhelming majority of cases. It also forces `striker_id`/`bowler_id` back to each ball's logged `batsman_id`/`bowler_id` before applying that ball during replay, making the reconstruction self-correcting against any drift rather than compounding errors across many balls.
- **A `'use server'` file may only export async functions** — every export from such a file becomes a server action reference, and Next.js rejects synchronous function exports at build time. `battingRoster`/`bowlingRoster` were initially written as local synchronous helpers inside `src/lib/actions/scoring.ts` and had to be moved into the pure `src/lib/cricket/scoring.ts` module instead (where they belong anyway, being pure functions with no I/O) once this constraint surfaced as a build failure (`'bowlingRoster' is defined but never used` was the actual lint error encountered, after a related but distinct mistake of leaving an unused import — the underlying async-only constraint is the thing to remember for any future server action file).
- The scoring console deliberately does NOT wrap run/extras/wicket buttons in a `<form>` — every action goes through `useTransition` and calls a server action directly from an `onClick` handler. A form-based approach would need a submit step or per-button hidden submit triggers; direct `useTransition` calls give the same pending-state UX (disabling buttons during the request) without that indirection, and matches the pattern already established for `TossForm`/`DeleteMatchButton` in Step 4.

## Step 6 Edge Cases & Notes
- The Realtime hook uses the browser client (createClient from lib/supabase/client.ts), NOT the server client — WebSocket connections must run client-side. The server component handles initial SSR fetch, the hook takes over for live updates.
- Map iteration with for...of requires ES2015+ target. The tsconfig here compiles to ES5 by default, so Map.entries() and Map.values() must use Array.from() explicitly rather than destructured for...of. Caught at build time as a TS error.
- The matches listing page shows live scores directly from live_data JSONB without extra queries — a key design payoff from the Step 1 decision to denormalize volatile state into live_data rather than requiring ball-log aggregation for every scoreboard render.
- 002_enable_realtime.sql must be run separately from 001_initial_schema.sql because the supabase_realtime publication already exists in hosted Supabase but the ADD TABLE statements need the tables to exist first (chicken-and-egg if run together in a single migration).
